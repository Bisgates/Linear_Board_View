# Linear Board — Tauri shell

Tauri 2.x wrapper around the existing Vite + React 18 board view. Lets the
project ship as a double-clickable macOS `.app` instead of running through a
local Vite dev server.

## Commands

| Command                        | What it does                                                                |
| ------------------------------ | --------------------------------------------------------------------------- |
| `npm run dev`                  | Original browser dev workflow. Vite + `linearApiPlugin` (Node HTTP routes + node-pty agent poller). Unchanged. |
| `npm run tauri:dev`            | Launches a native macOS window pointed at a Vite dev server on `:1420`. Hot reload works. |
| `npm run tauri:build`          | Produces `src-tauri/target/release/bundle/macos/Linear Board.app`.          |
| `npm run tauri:frontend-dev`   | Just the Vite half of `tauri:dev` (used internally by `beforeDevCommand`).  |
| `npm run tauri:frontend-build` | Just the Vite half of `tauri:build` (used internally).                      |

`tauri:dev` and `tauri:build` use a separate Vite config (`vite.tauri.config.ts`)
that intentionally **drops** the `linearApiPlugin`. Under Tauri those routes are
served by the Rust backend (`src-tauri/src/lib.rs`) instead — see below.

## Architecture

The frontend is the same React/TypeScript codebase that powers `npm run dev`.
At boot, `src/main.tsx` calls `installTauriBridge()` (in
`src/lib/tauriBridge.ts`).

- **Browser dev** (`npm run dev`): `installTauriBridge()` detects no Tauri
  runtime and returns `false`. Every `fetch("/api/...")` and
  `fetch("/data/issues.json")` hits the Vite plugin as before.
- **Tauri** (`tauri:dev` / `tauri:build`): the bridge patches `globalThis.fetch`
  to intercept `/api/*` and `/data/issues.json`. Bridged requests dispatch to
  Tauri commands defined in `src-tauri/src/lib.rs` and return synthetic
  `Response` objects — the rest of the frontend code is unaware of the
  switch.

### Storage

Boards, manifests, and the issue snapshot are persisted under
`~/Library/Application Support/com.han.linearboard/data/` with the same JSON
shapes the Vite plugin uses:

```
data/
  issues.json
  all_issues_board.json
  working_on/
    views.json
    wov_*.json
  custom/
    views.json
    cv_*.json
```

On first launch each manifest is initialised with one empty board. Data does
**not** migrate automatically from the browser dev workflow's
`public/data/`. To seed the Tauri app with your existing day/custom views,
copy `public/data/` into the Application Support directory above.

### Linear API

`@linear/sdk` runs **client-side** under Tauri. The Rust backend exposes one
read-only command (`read_linear_api_key`) that returns:

1. `$LINEAR_API_KEY` from the environment if set, else
2. the contents of `~/Library/Application Support/com.han.linearboard/linear_api_key.txt` if present, else
3. `null`.

To wire up your key:

```bash
mkdir -p "$HOME/Library/Application Support/com.han.linearboard"
echo "lin_api_xxx" > "$HOME/Library/Application Support/com.han.linearboard/linear_api_key.txt"
```

The Vite-dev workflow continues to read from the project root `.env` file via
`dotenv`.

## Current limitations

### Agent_tmp tab is disabled in Tauri builds

The Vite plugin spawns Claude code agents in PTYs via `node-pty` (see
`src/server/agentPoller.ts` and `src/server/agentSessions.ts`). That doesn't
exist in the Tauri Rust backend yet, so:

- `GET /api/agent/sessions` returns `{ ok: true, sessions: [] }`.
- `POST /api/agent/start` and `POST /api/agent/:id/stop` return a 503 with
  `"Agent management disabled in Tauri build (pending native pty migration)"`.
- The Agent_tmp tab still renders OPUS-team issue cards in read-only fashion
  and overlays a one-line banner explaining the situation. The card click and
  comment paths still work — only the start/stop agent buttons fail.

The native pty migration is tracked under a separate spike arc.

### No code signing / notarisation

The bundled `.app` is unsigned. macOS may refuse to open it on first launch —
right-click → Open the first time, or
`xattr -dr com.apple.quarantine "Linear Board.app"` to clear the quarantine
flag.

### Single architecture

The release build targets the host architecture only (arm64 on Apple Silicon
machines). Cross-arch bundling is not configured.

## File map

```
src-tauri/
  Cargo.toml             # Rust deps
  build.rs               # tauri-build hook
  tauri.conf.json        # window size, bundle metadata, devUrl
  capabilities/
    default.json         # core:default permission set
  icons/                 # auto-generated by `tauri icon`
  src/
    main.rs              # entry point — calls `lib::run`
    lib.rs               # all #[tauri::command]s: board store, snapshot, open_path, read_linear_api_key
```

Frontend additions:

```
src/lib/tauriBridge.ts   # fetch-shim + Linear SDK client-side wiring
src/main.tsx             # calls installTauriBridge() at boot
vite.tauri.config.ts     # Vite config used for Tauri (no linearApiPlugin)
```
